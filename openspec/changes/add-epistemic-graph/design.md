## Context

Exomem currently has strong retrieval and epistemic hygiene primitives: hybrid search, a wikilink graph lane, context packs, contradiction review, supersession frontmatter, source/evidence provenance, and audit/reconcile. The missing piece is a first-class graph substrate over the same owned files. Basic Memory's graph advantage comes from typed observations and relations that can be traversed directly; Exomem should match and exceed that by deriving a richer graph from its governed KB instead of requiring a new mandatory syntax or replacing Markdown as the source of truth.

The governing constraint is pure substrate: Exomem measures and surfaces; Claude/humans reason. Deterministic parsing, embeddings, OCR/ASR/CLIP, and frozen classifiers are allowed measurement. A server-side reasoning LLM is out of scope. Optional model-backed suggestion paths must be default-off, soft-failing, and propose-only.

## Goals / Non-Goals

**Goals:**
- Represent rich typed knowledge units and relationships across files, blocks, sources, evidence, entities, projects/cases, and media-derived segments.
- Keep Markdown and existing frontmatter canonical; the graph sidecar is rebuildable derived state.
- Expose graph context and relation suggestions through the single command registry across MCP, REST, and CLI.
- Let context packs use graph neighborhoods when explicitly requested and available, while preserving existing `find` behavior.
- Maintain graph freshness through writer hooks, watcher events, and reconcile, with bounded rebuild fallback.
- Make deterministic extraction useful before adding any optional model-backed suggestion path.

**Non-Goals:**
- No mandatory Basic Memory-style inline relation syntax.
- No server-side reasoning agent, auto-supersession, auto-decay, or authority/confidence floats on notes.
- No migration to a graph database as the canonical store.
- No change to default `find` ordering or `find(pack=false)` output.
- No required GPU, torch, or network dependency for graph indexing.

## Decisions

### Derived SQLite sidecar, not graph-as-storage

Create a rebuildable SQLite sidecar for graph state. Candidate implementation modules:
- `semantic_blocks.py` for deterministic block extraction.
- `epistemic_graph.py` for schema, indexing, lookup, and relation suggestion.
- Thin command leaves in the existing registry for `graph_context` and `suggest_relations`.

The sidecar should store nodes and edges with provenance, not truth claims:
- Nodes: stable key, kind, path, anchor/span, title, text excerpt, metadata JSON, source hash.
- Edges: source key, target key, relation type, origin (`frontmatter`, `wikilink`, `semantic_block`, `media_metadata`, `derived`, `suggested_response_only`), source path/span, reason/evidence JSON.
- Metadata: schema version, indexed scope, source freshness key, and rebuild timestamp.

Alternatives considered:
- Basic Memory-style SQL graph as the primary store: rejected because Exomem's moat is owned, editable Markdown with provenance and supersession.
- Inline typed relation syntax only: rejected because it creates adoption friction and misses relations already implied by frontmatter, sources, and existing notes.

### Semantic blocks are broad, structural, and source-spanned

Extract block nodes from ordinary Markdown before introducing optional syntax. The initial taxonomy should include `source`, `evidence`, `claim`, `finding`, `decision`, `assumption`, `constraint`, `risk`, `failure`, `experiment`, `result`, `pattern`, `requirement`, `action`, `entity`, `project`, `case`, `timeline_event`, and `media_segment`.

Deterministic extraction should recognize:
- Page type and frontmatter (`type`, `status`, `sources`, `supersedes`, `superseded_by`, `related`, project fields).
- Canonical sections such as `## Claim`, `## Findings`, `## Decision`, `## Risks`, `## Evidence`, `## Results`, `## Actions`, and known page-type templates.
- Wikilinks and Markdown links outside code blocks.
- Existing media metadata generated by extraction tools.

Unknown headings remain searchable content and do not become errors. Optional typed bullets can be added later, but the first implementation should win on zero-friction extraction from current KB pages.

### Relation types stay explicit and reviewable

The graph should support a broad relationship vocabulary, including `supports`, `contradicts`, `refines`, `duplicates`, `supersedes`, `derived_from`, `evidenced_by`, `depends_on`, `implements`, `mitigates`, `caused_by`, `blocks`, `answers`, `raises_question`, `used_for`, `observed_in`, `mentions`, `about_entity`, and `links_to`.

Only relations directly observed or deterministically derived from files are persisted in the sidecar. Relation suggestions are returned to the caller with evidence and method but do not mutate Markdown or the sidecar as accepted facts. If a human accepts a relation, it becomes durable only through an explicit note/edit/frontmatter update that the graph can parse on the next index pass.

### Graph context is bounded and read-only

`graph_context` should accept a path or query plus optional depth, relation-type filters, node-type filters, and caps. It returns the seed node(s), block nodes, related nodes, edges, provenance, and truncation notices. Traversal should default to shallow depth and explicit caps to avoid token blowups.

`suggest_relations` should accept a path or draft title/body and return candidate edges with method and evidence. Deterministic methods come first: shared sources, shared entities, co-citation, existing supersession fields, lexical block labels, wikilink neighborhoods, and embedding proximity when embeddings are available.

### Model-backed suggestions are optional measurement

Any NLI, polarity, claim classification, or other model-backed suggestion path must be:
- disabled by default;
- behind explicit env/config and optional extras;
- soft-failing when unavailable;
- labelled as measurement/proposal, never as reasoning;
- forbidden from directly writing notes, sidecar accepted edges, supersession fields, or ranking changes.

This keeps the server within the pure-substrate boundary while still leaving room for deeper offline measurement.

### Freshness integrates with existing watcher/reconcile

Graph indexing should reuse existing freshness concepts. Writer hooks and watcher events notify the graph index about changed/deleted Markdown paths. The graph index updates only affected rows when possible and falls back to rebuild when the sidecar is missing, schema-mismatched, or drifted. `reconcile` should repair graph drift the same way it repairs other derived indexes.

## Risks / Trade-offs

- [Risk] The taxonomy becomes too broad to implement cleanly. -> Mitigation: ship deterministic block extraction and core edge types first, then extend taxonomy through table-driven aliases and tests.
- [Risk] Graph context can become noisy or token-heavy. -> Mitigation: default shallow depth, relation/node filters, caps, and explicit truncation.
- [Risk] Optional model-backed suggestions are mistaken for truth. -> Mitigation: keep them default-off, response-only, labelled by method, and never durable without explicit user edit.
- [Risk] Incremental graph maintenance misses filesystem events. -> Mitigation: include schema/freshness keys, drift audit, and reconcile rebuild fallback.
- [Risk] Context packs change existing behavior. -> Mitigation: graph enrichment is opt-in and soft-failing; `find(pack=false)` and default pack behavior remain unchanged.

## Migration Plan

1. Add pure parser/index unit tests over temporary vault fixtures.
2. Implement semantic block extraction with no dependencies beyond existing Markdown/frontmatter helpers.
3. Implement the SQLite sidecar schema and full rebuild path.
4. Add graph context and suggestion leaves, then expose them through the command registry.
5. Wire graph enrichment into context packs behind explicit request and availability checks.
6. Add incremental freshness hooks, audit drift checks, and reconcile rebuild.
7. Add docs/scaffold updates only after command behavior is covered by tests.

Rollback is straightforward: disable graph enrichment and suggestion flags, ignore or delete the sidecar, and keep Markdown/search behavior unchanged.

## Open Questions

- Which relation aliases should be allowed in optional inline syntax, if any, after deterministic extraction lands?
